# 设计落地指南（Design 阶段）— 选型与参数规则

> 本文档覆盖 Design 阶段的**具体选型规则**：颜色与图标怎么选、字段类型怎么选、视图类型怎么选、自定义动作三类型、工作流触发/节点选型、图表 16 种类型、页面 6 种组件，以及跨模块闭环的具体接线。

---

## 零、颜色与图标通用法则

### 配色法则（icon_color / nav_color / 按钮等主题色，只能从这 9 色里选）

为应用、自定义动作、自定义页面**主题色**（design 里的 `app.icon_color` / `app.nav_color`、按钮 `color` 等）时，
**仅允许且必须使用以下 9 种 HAP 高端主题色**。任何其它 Hex（如纯红 `#FF0000`、纯蓝 `#0000FF`、纯绿 `#00FF00` 等）**严禁传入**：

| Hex 色值 | 视觉感知与推荐业务场景 |
| :--- | :--- |
| **`#A00416`** | **典雅红**。风险控制、警报、高优先级的核心金融或生命科学模块。 |
| **`#F21B65`** | **活力粉**。时尚、零售、社交娱乐、以客户体验为核心的协同板块。 |
| **`#FC532E`** | **温暖橙**。物流配送、现场服务、警示提醒、敏捷运营看板。 |
| **`#8E481B`** | **质感棕**。历史档案、物资仓储、重型设备管理、公共资源归档。 |
| **`#732ED1`** | **睿智紫**。人工智能、高新技术、战略规划、创意设计、前瞻性业务。 |
| **`#3054EB`** | **科技蓝**。通用稳重的主力配色——核心业务流水、综合看板、工作量统计。 |
| **`#21710F`** | **生态绿**。环境保护、供应链、农业技术、企业健康度。 |
| **`#4CAF50`** | **清新绿**。敏捷任务状态（正常/已完成）、日常审批、用户自助服务。 |
| **`#4051B6`** | **商务深蓝**。严谨的 HR 组织架构、合同档案、系统配置、权限后台。 |

**配色黄金法则**：颜色情感必须与所承载的业务实体高度契合（财务用科技蓝/温暖橙、异常告警用典雅红、仓储用质感棕）。

> 例外：选项字段/选项集的**状态色**（如 草稿=灰、进行中=蓝、已完成=绿、异常=红）按交通灯语义直觉取色即可，不受这 9 色限制——
> 这 9 色法则约束的是**应用/页面/按钮的主题色**，不是单选项的状态标记色。

### 图标法则（必须用 `hap icon search` 获取，严禁编造）

design 里任何 `icon` 字段（`app.icon`、`worksheet.icon`、`custom_page.icon`、按钮 `icon` 等）的值，**必须是真实存在的图标标识，不能凭空写一个名字**。

**正确做法**：生成 design 时，对每个要配图标的对象，用 `hap icon search` 一次堆上多个相关关键词（业务词、近义词、具象物件统统加上，多关键词取并集、命中越多排越前），从搜索结果中挑选一个：

```bash
hap icon search 会员 用户 成员 权益 皇冠 勋章
# Icon                    Keywords
# ----------------------  -------------
# sys_10_5_star           用户 评分
# sys_6_2_female_user     女性用户 用户 头像
# sys_6_3_user_male       男性用户 用户 头像
# sys_bleah_people        个人 头像 用户
# sys_medal2_object       奖章 勋章 荣誉
# sys_6_1_user_group      用户组 团队 群组
# ...
```

- 关键词宁多勿少：一个词搜不到不代表没有合适图标，换近义词、加具象物件词再搜。
- 命中不足 10 个时命令会自动随机补足推荐图标（JSON 输出中带 `suggested: true` 标记），可以直接从推荐里挑，也可以换关键词重搜。

- 从返回结果中选择最贴切的，用 **`Icon`** 列的值作为该对象的 `icon` 值。
- 宁可用一个稍泛的真实图标，**也绝不编造**库里没有的名字（编造会导致建出来图标缺失或报错）。
- **`validate` 会强制核验** design 里每一个 `icon` 是否真实存在：编造的名字（如 `sys_15_3_user`）会被判错，任一 icon 不通过则 validate 直接失败（exit 2），需改对后再 build。

**图标黄金法则**：

1. **语义匹配**：图标表意要能精准投射到该工作表/页面/按钮的业务范畴。
2. **零冗余（硬约束，会被 validate 强制查重）**：同一应用里**每张工作表、每个自定义页面的图标都必须互不相同**——两张表不能共用一个图标，表和页面之间也不能撞图标（它们并排出现在左侧导航，撞图标就视觉混淆）。这不是建议而是硬性规则：`validate` 会跨「工作表 + 自定义页面」查重，任一重复直接判错（exit 2），改对才能 build。
   - **怎么保证不撞**：逐表/逐页搜索图标时**换不同关键词**（别所有"单据/记录"类的表都用同一组词搜、再都取第 1 个），并**维护一份「本应用已用图标」清单**，选定一个就记下来、后面的对象避开它。
   - **例外**：自定义动作按钮、页面内组件的图标**不受此约束**（多个"新建/+"按钮复用同一图标是合理的），查重只针对导航级实体（工作表/页面）。
3. **严格受限**：绝不猜测或合成任何非常规图标名。

## 一、字段选型规则

为每个业务含义选对字段类型：

* 自由输入短文本 → `Text`
* 带格式长文本 → `RichText`
* 普通数值 → `Number`
* 金额 → `Currency`
* 固定单值枚举 → `SingleSelect`
* 固定多值枚举 → `MultipleSelect`
* 选项较多需搜索 → `Dropdown`
* 只要日期、年、年月 → `Date`
* 要具体时点 → `DateTime`
* 只要时间（不含日期）→ `Time`
* 单个是/否 → `Checkbox`（不要用 SingleSelect + 是/否）
* 1～10 级评分 → `Rating`
* 手机号 → `PhoneNumber`
* 座机/电话 → `LandlinePhone`
* 邮箱 → `Email`
* 定位坐标 → `Location`
* 省市区行政区域 → `Region`
* 负责人/审批人 → `Collaborator`
* 部门归属 → `Department`
* 组织角色 → `OrgRole`
* 数值计算（仅支持 +-*/() 数学运算）→ `Formula`
* 日期计算/差值/偏移 → `DateFormula`
* 指向另一张表的记录 → `Relation`
* 本表父子层级（上级任务/父分类）→ `Relation`（dataSource=selfRelation）
* 想把关联表的某列显示在本表 → `Lookup`
* 对明细/关联多条求和计数 → `Rollup`
* 一单挂多条明细 → `SubTable`
* 单据流水号 → `AutoNumber`
* 材料/图片/文件 → `Attachment`

### 关联字段规则

仅在以下情况建立 `Relation`：需要指向另一张表记录 / 需要与另一张表联动 / 需要基于另一张表筛选·统计·回写 / 需要表达本表父子层级。要在目标表反向看到则加 `two_way`。

### 误用边界（类型层）

* 状态、类型、优先级不要用 `Text`
* 负责人不要用姓名文本，优先用 `Collaborator`
* 不要把所有字段都做成 `Text`
* 不要仅因"查看方便"建立关联
* 不要过度设置必填字段

## 二、视图选型规则

每张工作表默认自带 `全部`（table）视图，无需创建。再按字段结构与角色需要增加第 2～5 个视图。

### 字段/业务驱动的视图（满足条件**必须**添加）

| 字段条件 | 必须添加的视图 | 说明 |
|---|---|---|
| 存在 selfRelation 或指向本表的 Relation | `hierarchy` | 父子树结构，Table 无法直观展示层级 |
| 存在状态/阶段类 SingleSelect（选项 ≥ 3） | `kanban` | 状态流转是看板核心场景 |
| 存在 Location 定位字段 | `map` | 地理分布只有地图能直观呈现 |
| 附件字段是表的核心内容（产品图、案例封面） | `gallery` | 图片化浏览是 Gallery 不可替代优势 |
| 业务语义是任务/排期，且有计划开始+结束日期 | `gantt` | 项目排期、任务计划需要时间轴 |
| 业务语义是排班/预约，且有时间段+人员/资源字段 | `resource` | 资源占用与时间冲突只有资源视图能呈现 |
| 参数配置/系统设置，通常只有 1 条记录 | `detail`（detail_mode=first） | 单记录无需列表 |

### 选型规则

* 批量浏览、筛选、编辑、导出 → `table`
* 当前人只看自己数据 → 增加筛选型 `table` 视图（如 `我的待处理`）
* 状态泳道、拖拽推进 → `kanban`
* 看某天有什么 → `calendar`
* 看任务起止与排期 → `gantt`
* 看图片内容（产品、案例、简历）→ `gallery`
* 看单条完整信息 → `detail`
* 看父子结构 → `hierarchy`
* 看地理分布 → `map`
* 看资源/人员占用与排班 → `resource`

### 误用边界

* 有日期字段，不代表一定建 `calendar`
* 有开始/结束日期，不代表一定建 `gantt`
* 有附件字段，不代表一定建 `gallery`
* 有关联字段，不代表一定建 `hierarchy`
* 不要因为"要分组"就一律用 `kanban`，很多场景更适合 `table` + 分组
* 每张工作表视图建议不超过 5 个

## 三、自定义动作选型

### 三种类型

1. **填写当前记录**（`update_record`）：把一组字段归类填写，方便集中操作。
   > 适用：需要用户**主动输入内容**才能推进流程。例：点击"更新图书信息"弹表单填写。
2. **新建关联记录**（`create_related`）：点击按钮新建一条关联记录。
   > 适用：在当前记录上**发起一件需要填写信息的新事务**。例：在"设备"上点"报修"，填"报修申请"。
3. **直接触发工作流**（`trigger_workflow`）：不弹表单，点击后由工作流自动处理。
   > 适用：业务推进**不需要用户输入**。例：借阅记录上的"归还"——状态自动改、归还日期写今天、库存 +1。

### 挂载规则

动作必须挂在**动作发起的数据源头表**上，不挂目标表（发起借阅挂图书表、登记归还挂借阅记录表）。
> 即使不触发工作流，也应为高频操作（填写记录、新建关联）尽量提供动作以提高便利性。

### 误用边界

* **严禁在没有 Relation 字段的两张表之间设计 `create_related`。** 规划 `create_related` 时**必须同时提供用于物理关联的具体 Relation 字段名**（该字段须显式声明在源工作表 `fields` 中）。若两表间无 Relation，则要么：(1) 在源表字段列表显式追加 Relation 字段并在动作中引用；(2) 或改用 `trigger_workflow`，由后台工作流自动创建对应记录。
* 不要把"新建关联记录"类动作挂在目标表上
* 不要让需要人工输入的场景用 `trigger_workflow`
* 不要把本可自动完成的动作做成 `update_record/create_related`
* 系统可自动确定的值（今天、当前用户、固定状态）不应要求用户手填
* 不需要人工点击的场景 → 改用自动化工作流，不做自定义动作

## 四、自动化工作流选型

### 触发方式选型

> ⛔ **优先级原则（先读，再选）：自动触发为主，按钮触发为辅。**
> 「自动化工作流」的价值就在**自动**——能由数据自己驱动的，就别要求人去点按钮。选触发方式时按下面顺序问，命中即停：
> 1. 这件事**该在记录新增/某字段变化时自动发生**吗？（状态同步、数据联动、统计回写、新建即通知）→ 用 `record_create` / `record_update` / `record_create_or_update`。
> 2. 这件事**该在某个日期到/前/后自动发生**吗？（到期提醒、超期标记、续费预警）→ 用 `date`。
> 3. 这件事**该按固定周期自动跑**吗？（每日逾期巡检、月度盘点提醒、周期汇总）→ 用 `scheduled`。
> 4. **只有**当这件事必须由人**主动决定何时发起**、系统无法自行判断时机时（提交审批、确认收货、发起退款/作废）→ 才用 `button`（配 `trigger_workflow` 自定义动作）。
>
> 换句话说：`button` 是兜底选项，不是默认选项。一个应用如果**绝大多数工作流都是 button 触发**，几乎一定是设计错了——把本该自动跑的流程错挂成了按钮。状态流转、库存增减、时效预警、数据联动这些都应当是自动触发。

| 触发方式 | 适用场景 | 典型应用 |
|----------|----------|----------|
| 记录触发（`record_create` / `record_update` / `record_create_or_update`） | 记录新增/编辑/新增或编辑后立即触发 | 通知推送、状态同步、数据联动 |
| 按日期字段（`date`） | 基于记录中某日期字段相对触发 | 到期提醒、定期跟进、续费预警 |
| 定时（`scheduled`） | 固定周期执行 | 逾期检查、周期汇总、盘点提醒 |
| 按钮触发（`button`，由自定义动作触发） | **仅**用户必须主动发起、系统无法自判时机时 | 审批提交、确认归还、发起退款 |

### 动作节点选型

* 生成新单据/流水 → 新增记录 (`create_record`)
* 推进状态/自动赋值/联动回写 → 更新记录 (`update_record`)
* 删除特定记录 → 删除记录 (`delete_record`)（⚠️ 仅必要时用，优先"更新状态为已作废"）
* 获取其他表数据 → 查询单条 (`get_single`) / 查询多条 (`get_multiple`) / 取触发记录的关联子表多条 (`get_relation_records`)
* 按条件批量更新（扣减/累加/回写）→ 查询并更新 (`query_update`)
* 对查询结果统计 → 汇总 (`rollup`)（按 `mode`：工作表汇总 worksheet / 对象计数 object）
* 动态计算数值或日期 → 计算 (`compute`)（按 `mode`：数值公式 number / 日期公式 date / 日期差 date_diff / 函数 function）
* 触达相关人员 → 站内通知 (`notice`) / 抄送 (`cc`) / 发送邮件 (`send_email`)
* 引入人工决策与流转 → 审批业务块 (`approval_block`)（从发起审批开始，内部可含审批 `approve`、填写 `fill_in`、抄送）
* 针对多条记录循环执行 → 子流程 (`sub_process`)（常配查询多条/取关联多条，遍历逐个处理）
* 按条件分流 → 条件分支 (`branch`)（默认唯一分支 `mode: exclusive`，只走第一个匹配路径）

> 节点的 config 键、值引用、审批人种类等隐藏规则**务必先读** [workflow-gotchas.md](workflow-gotchas.md)。

### 误用边界

* 不要用 `scheduled` 做实时通知（应用记录触发）
* 不要把高频变化规则写死在节点里，优先用字段值 + 条件分支
* 不要给所有流程都套审批
* 不要只发通知不写回数据
* **不要把本该自动触发的流程错挂成按钮**——状态流转、库存增减、时效预警、数据联动、新建即通知都应是 record/date/scheduled 自动触发；`button` 仅限「提交/确认/发起」这类必须人工主动发起的动作。一个应用里 button 流程占多数 = 设计跑偏。
* 删除记录要谨慎，很多场景更适合"更新状态为已作废"
* 含有不确定人工判断的步骤（要人来拍板的）才用自定义动作让人决策——但这指的是**人工决策点**，不是"凡是动作都做成按钮"

## 五、统计图表选型

> **支持全部 16 种 `report_type`**：`number` / `bar` / `line` / `pie` / `pivot` / `funnel` / `radar` / `bidirectional_bar` / `ranking` / `scatter` / `wordcloud` / `world_map` / `dual_axes` / `gauge` / `progress` / `region_map`。

### 选型规则

* 看关键数字（单值指标，无维度）→ `number`（数值卡）
* 看完成率/目标达成 → `progress`（进度，值 + `target`）/ `gauge`（仪表盘，值 + 可选 `max_field`）
* 看类别对比/名称较长的对比 → `bar`
* 看排行榜/TopN → `ranking`
* 看时间趋势 → `line`
* 看双指标趋势（左右两轴）→ `dual_axes`（左 `metrics` + 右 `right_metrics`）
* 看占比构成 → `pie`
* 看阶段转化 → `funnel`
* 看多指标综合比较 → `radar`
* 看两组指标左右对称对比 → `bidirectional_bar`（1 维度 + 2 个值）
* 看两个数值变量关系 → `scatter`（1 维度 + 2 个值）
* 看文本热点/关键词分布 → `wordcloud`
* 看地区分布（国内省市区）→ `region_map`（维度取 Region 字段 + `map_level`）
* 看地区分布（全球国家）→ `world_map`（维度取国家字段）
* 看多维交叉分析（行 × 列 × 值）→ `pivot`（用 `rows`/`columns`/`metrics`）

### 每个图表必须先明确

回答什么业务问题 / 指标是什么 / 按什么维度看 / 为什么值得进 dashboard 首页。

### 误用边界

* 不要多个图表回答同一个问题
* 不要堆过多 `number` 或 `pie`
* `pie` 类别过多时改用 `bar`
* 没有明确阶段顺序不要用 `funnel`
* 没有连续时间维度不要用 `line`
* 每个 dashboard 建议 6~10 个图表；只有确需多维交叉时才用 `pivot`

## 六、自定义页面组件

自定义页面支持以下 **6 种组件**（`type` 的英文值），其它（轮播图、标题区、标签页、卡片、图片等）**不支持**：

1. **图表 (`chart`)**：`report_type` 取上面图表选型中的各类型。聚合展示与分析（`dashboard` 核心）。
2. **快捷按钮 (`button`)**：每个按钮配置 `create_record` / `open_view` / `open_page` / `open_link`。工作台与任务中心（`workspace` 核心）。
3. **内嵌视图 (`view`)**：把工作表某个视图直接嵌入页面。
4. **富文本 (`rich_text`)**：展示操作规范、规章制度等 HTML 内容。
5. **网页内嵌 (`embed_url`)**：以 iframe 嵌入外部网址。
6. **全局筛选条 (`filter`)**：放页面顶部，按字段联动页面内多个图表统一筛选。

### 误用边界

- 不要混淆组件定位（如在 workspace 堆折线图，或在 dashboard 放过多操作按钮）
- 不同页面应有明确分工，不重复展示相同数据或入口

## 七、跨模块闭环（具体接线示例）

设计时必须确保以下闭环完整（Plan 阶段已要求闭环**存在**，此处给出落地接线）：

### 1. 工作流数据闭环

每条流程优先形成：①更新主表状态 → ②更新关联表统计/状态 → ③新增流水/日志 → ④发送通知。只有通知、没有数据写回或留痕，闭环不足。

### 2. 视图 ↔ 字段

视图筛选条件必须有对应字段和选项值支撑。
* ✅ 规划"超期事项(table)"视图，状态字段含"已超期"：`事项状态(SingleSelect:待分解/办理中/已完成/已超期/已归档)`
* ❌ 规划"超期事项"视图，但状态字段缺"已超期"选项——视图无法筛选

### 3. 字段 ↔ 工作流（时效状态）

状态里凡有系统自动判定的选项（已超期/已逾期/已过期），必须有工作流自动标记。
* ✅ 状态有"已超期"，同时有 `date`（按日期字段）触发流：完成时限到达时，若未完成则更新为已超期并通知责任人
* ❌ 状态有"已超期"但无任何工作流——该选项永不被设置，视图始终为空
* 典型实现：`date` 触发（时限到达检查未完成则标记）/ `scheduled` 触发（每日检查所有未完成记录）

### 4. 自定义动作 ↔ 字段 ↔ 工作流

动作前置条件和执行结果都要有字段支撑，状态变更要有工作流执行。
* ✅ "归档"按钮启用条件"事项状态=已完成"，点击触发工作流更新为"已归档"
* ❌ "归档"按钮无启用条件，任何状态都可点——未完成的也能被归档

### 5. 自定义页面 ↔ 视图/字段

工作台内嵌视图、看板图表都依赖字段完整。
* ✅ 仪表盘"超期事项数(number)"指向事项表，筛选"状态=已超期"，字段和工作流均配套
* ❌ 仪表盘有"超期事项数"，但状态字段无"已超期"、也无工作流标记——图表始终显示 0
