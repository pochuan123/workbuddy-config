# 方案设计指南（Plan 阶段）

> 本文档指导你在 **Plan 阶段**把业务需求想成一套**专业、完整、闭环**的应用方案。
> 这里只讲**设计思想**——建哪些对象、为谁建、为什么这样组织。

---

## 零、应用命名规范

应用名称代表整个系统的业务边界与专业度，应极致简洁、业务场景聚焦、富现代感：

1. **剔除冗余后缀**：**严禁**使用"系统""平台""软件""工具""应用""表""系统后台"等传统而笨重的后缀。
   - ❌ 传统冗长：`SaaS 客户订阅与服务生命周期管理系统`、`重型设备维保管理平台`
   - ✅ 现代极简：`SaaS 客户订阅与服务`、`设备维保与健康监测`、`图书借阅大厅`
2. **场景聚焦与字数限制**：建议控制在 **2~8 个字**，精准突显核心业务实体与动作概念。

---

## 一、应用设计顺序（角色驱动）

设计一个 HAP 应用，按以下顺序推进，每一步都依赖上一步的结论：

| 步骤 | 模块 | 核心问题 |
|------|------|----------|
| 1 | **角色权限** | 谁使用这个应用？各角色职责和权限边界是什么？ |
| 2 | **工作表** | 需要管理哪些业务对象？数据模型如何支撑各角色的工作？ |
| 3 | **自动化工作流** | 哪些流程节点可以自动化？哪些需要人工触发？ |
| 4 | **自定义页面** | 管理者需要看什么数据看板？一线操作者需要什么工作台入口？ |
| 5 | **导航分组** | 如何把上述内容组织成用户友好的入口层级？ |
| 6 | **外部门户** | 是否有外部用户需要受限访问？ |
| 7 | **AI 助手** | 是否有明确的智能查询或数据分析需求？ |

> **全局原则——以角色驱动设计**：角色是贯穿整个应用的设计主线。每个视图、自定义页面、工作流、自定义动作都应先明确"服务于哪个角色"，再决定内容：
> - **视图**：同一张工作表，不同角色看到不同的视图（数据范围、筛选条件、展示字段各异）
> - **自定义页面**：不同角色拥有各自的仪表盘或工作台（管理者看经营分析，操作者看待办任务）
> - **自定义动作**：按钮应只对有权操作的角色可见
> - **工作流通知**：通知对象应精确到角色，避免全员广播

---

## 二、角色权限设计

确定谁使用应用、能看到什么、操作什么。角色基于 RBAC 思想设计。

> **先想清楚「有哪些业务责任」，再决定「需要哪些数据权限」。**
> 角色不是组织架构，也不是岗位名单，而是一组可复用的业务权限能力。HAP 支持一个用户拥有多个角色，应优先拆分清晰的业务角色，再通过多角色组合覆盖复合岗位。

### 系统预设角色（不要重复创建）

| 预设角色 | 权限范围 |
| ---- | ---- |
| 应用管理员 | 所有权限（数据 + 应用配置） |
| 运营者 | 所有数据权限 + 角色/用户管理 |
| 搭建者 | 应用编辑权限（结构搭建） |

设计时只需创建**业务角色**，无需创建"系统管理员""超级管理员""平台管理员"等系统级角色。

### 角色设计原则

**1. 按"业务职责"而非"组织层级"设计**：角色名应对应实际工作职能。

| 推荐 | 不推荐 | 原因 |
| ---- | ---- | ---- |
| 借阅人员、图书管理员 | 普通员工、部门成员 | 层级描述，权限边界不清晰 |
| 采购专员、财务审核 | 采购部、财务部 | 部门不等于角色 |
| 销售代表、风控专员 | 业务人员、后台人员 | 职责过泛，无法指导权限设计 |
| 维修厂 | 外部用户 | 应按外部协作职责命名 |

**2. 按"权限差异"决定是否拆分角色**：只有在以下任一方面存在明显差异时才拆分——能看到的数据范围、能操作的工作表、能执行的动作、参与的审批节点、使用的视图/页面入口、是否外部用户。差异不明显则合并。

**3. 利用"多角色组合"，避免过度复合角色**：

| 不推荐 | 推荐 |
| ---- | ---- |
| 销售主管兼客户经理 | 销售主管 + 客户经理 |
| 采购审批兼财务审核 | 采购审批人 + 财务审核 |
| 仓管兼资产管理员 | 仓管人员 + 资产管理员 |

### 角色完整性检查

生成业务角色时，从以下角度检查是否遗漏（不是每个应用都全有，按业务流程判断）：

| 角色类型 | 说明 | 示例 |
| ---- | ---- | ---- |
| 一线操作角色 | 日常录入、跟进、处理业务数据 | 销售代表、咨询员、仓管人员 |
| 业务管理角色 | 团队监督、进度跟踪、异常处理 | 销售主管、设施管理者、车队管理员 |
| 审批审核角色 | 关键节点审批、复核、确认 | 财务审核、风控专员、采购审批人 |
| 专业支持角色 | 某类专业业务处理 | 法务专员、评估员、财务专员 |
| 外部协作角色 | 通过门户或受限入口参与业务 | 维修厂、供应商、客户、合作机构 |
| 只读分析角色 | 查看统计/报表/驾驶舱，不直接操作数据 | 管理层、法人管理者、观察者 |

### 误用边界

- 不要创建"系统管理员""超级管理员"等系统预设角色
- 不要按部门名直接建角色，除非该部门本身就是明确的数据权限边界
- 不要把权限完全相同的岗位拆成多个角色
- 不要为某个人的复合职责创建混合角色，应用多角色组合
- 不要只设计一线录入角色——还要检查审批、监督、分析、外部协作角色
- 不要把"管理层"默认设计成可操作角色，通常应优先作为只读分析角色

## 三、工作表建模

数据是应用的基础，工作表设计决定整个应用的信息结构。

### 两类工作表

| 类型 | 特征 | 典型例子 |
|------|------|----------|
| **基础资料类**（Master Data） | 主数据，相对稳定，被其他表引用 | 客户、图书、设备、员工、商品 |
| **业务过程类**（Transactional Data） | 流程记录，随业务持续产生 | 订单、申请、借阅记录、巡检记录 |

### 命名准则

- **基础资料类**：业务对象名（名词单数）。✅ `图书`、`客户`　❌ `图书列表`、`客户信息表`
- **业务过程类**：对象 + 动作名词。✅ `借阅记录`、`采购申请`、`维修工单`

### 建表判断

**建表前问自己**：这个对象需要独立的生命周期管理吗（需增删改查）？
- **是** → 独立建表，通过关联引用
- **否**（只是分类标签）→ 用单选/多选字段，不建表

### 核心原则

- **单一数据源**：同一业务对象只建一张表，用字段区分状态，用视图区分角色视角，用权限控制访问范围
- **状态机思维**：业务对象的不同阶段是同一张表里的状态字段。"待审批"和"已审批"是状态值，不是两张表

### 误用边界

- 不要按角色拆表——权限差异通过角色配置解决
- 不要把流程阶段拆成多张独立表
- 分类不需要单独管理时用单选字段，不建表

---

## 四、字段完整性

设计一张工作表字段时，按以下 7 个维度依次补齐，**每个维度至少 1 个字段**：

| 维度 | 说明 | 典型字段示例 |
|---|---|---|
| ① 主标识 | 名称、编号、标题——一眼识别这条记录 | 书名、订单编号、工单标题 |
| ② 核心业务属性 | 体现行业特征的**核心描述信息** | ISBN、出版社、页数、作者（图书）；规格、品牌、产地（商品） |
| ③ 状态与分类 | 状态流转、类型分类、优先级 | 借阅状态、图书分类、紧急程度 |
| ④ 时间维度 | 创建时间、截止日期、完成时间、有效期 | 借出日期、应还日期、入库日期 |
| ⑤ 责任与协作 | 负责人、创建人、审批人、操作人 | 经办人、管理员、借阅人、审批人 |
| ⑥ 数量与金额 | 数量、金额、统计值——支撑工作流加减运算 | 库存数量、单价、总金额、累计次数 |
| ⑦ 说明与凭证 | 备注、附件、图片、富文本说明 | 图书封面、备注、合同附件 |

> ⚠️ **维度 ② 最容易被忽略**。很多设计只有名称+状态+时间，缺少真正体现业务对象特征的属性字段。

### 按业务场景补齐易遗漏字段

- **主数据/资料类**（客户、图书、设备）：图片/封面、简介/描述、所在位置/地址、启用状态、创建时间、**行业特征属性**（ISBN/出版社、规格/品牌、型号/序列号/购入日期）。
- **单据/记录类**（订单、工单、入库单）：单据编号、操作人、操作时间、**处理状态（必须包含完整状态链，覆盖从创建到结束的所有状态）**、来源/渠道。
- **涉及库存/数量**：总量、已用量/在借量、可用量/剩余量——确保能支撑工作流加减（借出 -1、归还 +1）。
- **涉及审批/流转**：审批状态、审批人、审批时间、审批意见。

> 不要只生成总览里提到的几个字段就停止——要按上述维度补齐业务骨架。

## 五、视图规划（按角色，而非按字段堆砌）

每张工作表默认自带一个 `全部` 视图，无需规划。再根据业务和角色查看需要补充高价值视图：

- 一线操作者 → "我的待处理""我负责的"等只看自己相关数据的筛选视图
- 业务管理者 → "全部记录""团队待办"等全局视图，用于监控和分配
- 审批角色 → "待我审批"视图
- 只读分析角色 → 通常按需分配已有视图的查看权限即可，不需专属视图

> 视图要服务于明确的角色视角；同一张表不同角色看不同视图，而非按角色拆表。

---

## 六、自定义动作规划

工作表记录上的操作按钮（自定义动作）是串联业务流程的核心**人工入口**。规划时关注三件事：

1. **这一步要不要人工**：需要用户主动输入内容/发起事务 → 适合做按钮；系统可自动确定的值（今天、当前用户、固定状态）不应要求人工手填，应交给工作流。
2. **挂在哪张表**：动作必须挂在**动作发起的数据源头表**上，不挂目标表。
   - 发起借阅 → 挂图书表；发起工单 → 挂客户表；登记归还 → 挂借阅记录表。
3. **是否需要配套工作流**：需要后台自动处理（改状态、扣减库存、发通知）的按钮，必须同步规划对应工作流。

> 动作的三种类型与"新建关联记录必须有 Relation 字段"等具体约束在 Design 阶段处理，见 [design_guide.md](design_guide.md) / [custom-actions.md](custom-actions.md)。

## 七、自动化工作流规划

> **核心原则**：先理清业务流程，再考虑自动化。自动化只是加速，不能替代对流程本身的设计。

规划时确定每条流程的**触发时机**（概念层，code 与节点选型见 Design 阶段）：

- **记录触发**：记录新增/编辑后立即触发——通知推送、状态同步、数据联动
- **按日期**：基于记录中某日期字段相对触发——到期提醒、续费预警
- **定时**：固定周期执行——逾期检查、周期汇总、盘点提醒
- **按钮触发**：用户点击自定义动作后触发——审批提交、确认归还

### 误用边界（流程设计层）

- 不要用定时做实时通知（应用记录触发）
- 不要给所有流程都套审批
- 不要只发通知不写回数据
- 删除记录要谨慎，很多场景更适合"更新状态为已作废"
- 含有不确定人工判断的步骤不要自动化，用自定义动作让人来决策

## 八、统计看板与自定义页面规划

> 图表的 16 种类型选型、页面 6 种组件等具体规则属于 Design 阶段，见 [design_guide.md](design_guide.md) / [custom-pages.md](custom-pages.md)。本阶段只规划"页面服务于谁、回答什么问题"。

### 页面两种定位

| 页面类型 | 服务对象 | 内容侧重 |
|----------|----------|----------|
| **dashboard**（数据看板） | 需要看数据做判断的角色 | 趋势对比、多维分析、聚合统计 |
| **workspace**（工作台） | 需要高频操作的角色 | 当前状态、待办处理、高频操作入口 |

- 管理者/决策者 → 经营分析仪表盘（核心）+ 管理工作台（审批、督办入口）
- 一线操作者 → 操作工作台（核心）+ 业务仪表盘（个人绩效、待办统计）
- 角色关注内容不同则分别建独立页面；完全一致才共享同一页面

### 看板优先回答的问题

一个 dashboard 优先覆盖：①核心指标总览 ②时间趋势变化 ③结构占比或排行对比。每个图表都要能回答一个具体业务问题，不堆砌、不重复。

## 九、导航分组

导航结构决定用户进入应用后的第一印象和操作路径。

> **核心原则**：以用户的业务任务和使用场景为主线组织分组，兼顾核心业务对象，入口清晰、层级稳定。

| 分组位置 | 推荐内容 | 说明 |
|----------|----------|------|
| 1. 总览/首页 | 自定义页面（Dashboard/Workspace）、AI 助手 | 一眼掌握全局状态和核心待办 |
| 2. 核心业务 | 高频操作工作表 | 借阅管理、订单处理、工单跟进 |
| 3. 基础资料 | 主数据工作表 | 低频，放靠后 |
| 4. 配置与管理 | 分类维护表、基础数据表 | 极低频，管理员维护 |

- 组名 2~4 字，简洁明确，同层级保持一致抽象维度
- 分组通常 3~5 个，每组 2~5 个项目
- 禁止单项分组——只有 1 项时并入相邻分组

## 十、AI 助手规划

基于应用内数据提供智能问答、数据查询和分析建议。

| 适合 | 不适合 |
|------|--------|
| 数据查询："本月库存还剩多少？" | 替代工作流执行自动化操作 |
| 状态查询："我的申请审批到哪步了？" | 代替人工完成界面操作 |
| 辅助分析："哪类图书借阅量最高？" | — |

> 仅在有明确查询或分析需求时规划，不强制每个应用都配。

---

## 十一、跨模块闭环（方案必须自洽）

视图、字段、工作流、自定义动作之间存在强依赖，任何一环缺失都会让功能失效。方案阶段就要确保以下闭环**存在**（具体接线示例见 [design_guide.md](design_guide.md)）：

1. **工作流数据闭环**：每条流程优先形成"更新主表状态 → 更新关联表统计/状态 → 新增流水/日志 → 发送通知"。只发通知、不写回数据通常闭环不足。
2. **视图 ↔ 字段**：视图的筛选条件必须有对应字段和选项值支撑（规划了"超期事项"视图，状态字段就必须有"已超期"选项）。
3. **字段 ↔ 工作流（时效状态）**：状态里凡有系统自动判定的选项（已超期/已逾期/已过期），就必须配一条按日期或定时触发的工作流去自动标记——否则该选项永不被设置，视图永远为空。
4. **自定义动作 ↔ 字段 ↔ 工作流**：动作的启用前置条件和执行结果都要有字段支撑，状态变更要有工作流执行。
5. **自定义页面 ↔ 视图/字段**：工作台内嵌视图、看板图表都依赖字段完整——图表筛选的状态选项必须真实存在且有工作流维护。

---

## 产出质量要求

Plan 阶段的方案应满足：

- **角色**：覆盖一线/管理/审批/支持/外部/只读各类型，无遗漏
- **字段**：按 7 个维度补齐业务骨架，不只写总览提到的几个
- **视图**：按角色规划高价值视角
- **看板**：每个看板/图表都回答一个具体业务问题
- **工作流**：每条流程尽量形成数据闭环
- **自定义动作**：先判断是否真的需要人工点击与输入
- **闭环**：上述跨模块闭环全部自洽，没有"空选项""空视图""空图表"

> 方案确认后进入 Design 阶段，再对照 [design_guide.md](design_guide.md) 与各模块专题文档落实具体参数。
